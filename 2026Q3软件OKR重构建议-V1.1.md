# 2026 Q3 软件团队 OKR 重构建议 V1.1

生成日期：2026-06-09  
版本：V1.1  
范围：仅聚焦 Software；O1 质量方向基本沿用当前框架，重点重构 O2/O3。  
输入来源：当前 `Engineering OKRs` Google Doc、历史 `unitx-okr-diagnostic-report.md`、`2026Q3软件OKR重构建议.md`。  
边界：本文件是本地草案，不写回 Google Doc；`X/Y/Z` 需要后续用 Jira、release report、现场问题台账、项目阶段台账、FAE/Service 验证记录补齐。

## 1. V1.1 核心判断

O1 质量方向没有大问题，可以保留现有框架，只需要把 IPC 卡死类问题写成可验证的稳定性 KR。

真正需要重构的是 O2 和 O3。当前 Google Doc 中：

- O2: Reduce IPC BOM for EOL product without sacrificing reliability
- O3: Ensure a successful launch of smart camera

这两个 Objective 都太像单点项目。它们没有覆盖 Software Q3 需要承担的完整产品化和交付能力，也不方便继续拆给下一层团队。

V1.1 建议把 O2/O3 分成两类：

| Objective | 分类 | 覆盖事项 | 不建议的写法 |
|---|---|---|---|
| O2 | 产品化与市场验证 | 智能相机、标准机 | 只写 Smart Camera launch |
| O3 | 规模化交付与成本效率 | IPC 降本、workflow 优化、5.x readiness、标准交付包 | 只写 IPC BOM costdown |

这样分类后，四个必须覆盖的事项都有位置：

1. 智能相机：放入 O2，作为新产品产品化路径。
2. 标准机：放入 O2，作为标准方案产品化路径。
3. IPC 降本：放入 O3，作为单位交付成本和标准配置效率。
4. workflow 优化：放入 O3，作为 FAE/Service 自助交付和配置效率。

建议额外放入 O3 的事项：

- 5.x/EOL readiness：没有 readiness，IPC 降本和 workflow 优化都很难进入现场。
- 标准交付包：把 workflow、配置、日志、FAQ、版本约束打包，才能让 FAE/Service 独立使用。
- 部署诊断能力：不一定作为独立 KR，但应作为 O3 initiative，因为它直接影响交付阻塞天数。

## 2. 推荐 Software OKR V1.1

### O1. Improve EOL software quality and reliability

建议保留当前 O1，不在 V1.1 中大改。只建议把第三条 KR 从动作改成结果。

```markdown
O1: Improve EOL software quality and reliability
- KR1: Reduce P0/P1/P2 bugs in new EOL software releases reported by field (escaped QA) to X/Y/Z per release. [June]
- KR2: Reduce P0/P1/P2 bugs in new EOL software builds reported by QA (escaped Dev) to X/Y/Z per release. [Kai]
- KR3: Close IPC freeze P0/P1 stability risks: known IPC freeze issues have 100% root-cause classification and fix/workaround; target release passes X-hour stability validation on target IPC configurations; no recurrence in validation. [Colin]
```

说明：

- O1 的重点是质量和严重稳定性，不承载 IPC 降本。
- IPC 卡死可以留在 O1，因为它是 P0/P1 稳定性风险。
- IPC BOM 降本应进入 O3，因为它是成本效率和配置可靠性问题。

### O2. 推动智能相机和标准机完成产品化验证

分类：产品化与市场验证。

战略意图：Software 不应只承诺“支持 Smart Camera launch”，而应承诺智能相机和标准机这两条产品线的软件产品化结果：是否能在目标场景跑通 E2E，是否能形成可复用软件包，是否能被 FAE/Service 按标准流程交付。

| KR | 类型 | 建议 Owner | 指标定义 | 覆盖事项 | 数据源 |
|---|---|---|---|---|---|
| KR1. 智能相机在 `X` 个目标客户或标准应用场景完成软件 E2E 验证，并达到约定验收指标 | Committed | Yuanhang + Software | E2E 包括安装、配置、采图、推理/规则、结果输出、日志诊断；验收指标需在 Q3 前两周冻结 | 智能相机 | UAT report、客户/内部验收记录、release notes |
| KR2. 标准机在 `X` 个目标标准应用场景完成软件方案验证，证明相对定制交付在配置时间、稳定性或复用性上有明确改善 | Committed | Product + Software | 标准机不只看功能完成，要看是否减少项目制交付依赖；至少选择 `X` 个高频/高价值场景 | 标准机 | 标准场景清单、benchmark、FAE 验证记录 |
| KR3. 智能相机和标准机的软件 release package 覆盖安装、配置、版本约束、日志诊断、rollback/known issues，release readiness 达到 100% | Committed | Software PM/Lead | readiness checklist 必须可被 Product/Service 共同验收；未通过项需要 risk acceptance | 智能相机、标准机 | readiness checklist、release review |
| KR4. FAE/Service 在无研发现场支持下完成 `X` 次智能相机或标准机部署演练/准真实交付，成功率达到 `Y%` | Aspirational | Service + Software | 从开箱/安装到配置/运行/问题记录完整走通；失败项进入 blocker list | 智能相机、标准机 | deployment rehearsal、Service feedback、blocker list |

O2 的 initiatives 可以包括：

- 智能相机分类/OCR E2E workflow。
- 智能相机软件安装、配置、日志和诊断工具。
- 标准机目标场景选择和标准软件配置模板。
- 智能相机/标准机 release readiness checklist。
- FAE/Service 部署演练和 blocker review。

不建议放入 O2 的内容：

- IPC BOM costdown。它不是产品化验证，而是成本效率和配置可靠性。
- 一般质量指标。O1 已覆盖 escaped QA、escaped Dev、IPC 卡死等质量问题。
- 纯 roadmap/文档输出。除非它能证明产品化验证通过。

### O3. 提升 EOL 平台的规模化交付能力和成本效率

分类：规模化交付与成本效率。

战略意图：O3 应该承接 Software 对规模化交付的贡献。IPC 降本和 workflow 优化都不是孤立项目，本质上都服务于同一件事：用更低成本、更少研发介入、更标准化的软件能力交付 EOL 项目。

| KR | 类型 | 建议 Owner | 指标定义 | 覆盖事项 | 数据源 |
|---|---|---|---|---|---|
| KR1. 降本 IPC 目标配置通过 EOL 软件性能、稳定性和兼容性验证，不牺牲关键场景可靠性 | Committed | Nero + Software + QA | 与基线 IPC 配置对比；覆盖吞吐、延迟、资源占用、长稳、关键 workflow；不得引入 P0/P1 blocker | IPC 降本 | benchmark、长稳测试、配置矩阵、Jira |
| KR2. 降本 IPC 配置在 `X` 个真实或准真实 EOL 场景完成验证，进入标准配置或明确 Go/No-go | Committed | Nero + Product + Software | 不只做实验室测试；必须明确适用场景、禁用场景、版本约束和风险 | IPC 降本、标准机 | 场景验证报告、readiness checklist |
| KR3. Workflow 优化覆盖 Top `X` 个高频 EOL 配置/检测流程，使 FAE/Service 独立完成配置的成功率达到 `Y%`，配置耗时降低 `Z%` | Committed/Aspirational | Software + Product | 重点从“完成功能”改成“减少交付依赖”；workflow 应覆盖标准机和通用 EOL 场景 | workflow 优化、标准机 | workflow UAT、deployment rehearsal、Service feedback |
| KR4. 5.x/EOL 标准交付包覆盖目标新部署/升级项目 100%，进入现场前未关闭 readiness blocker 均有风险接受记录 | Committed | Software PM/Lead | 标准交付包包括版本、配置、workflow 模板、IPC 配置规则、日志诊断、rollback、known issues | workflow 优化、IPC 降本、5.x readiness | readiness checklist、release package、project tracker |

O3 的 initiatives 可以包括：

- IPC 降本配置 benchmark：吞吐、延迟、GPU/CPU/memory、长稳。
- IPC 标准配置矩阵和适用边界。
- Workflow 模板化：标准检测流程、配置参数、日志输出、异常处理。
- 5.x/EOL deployment readiness checklist。
- 标准交付包：配置模板、版本约束、FAQ、rollback、known issues。
- 部署诊断能力：日志采集、健康检查、问题定位脚本或 dashboard。

O3 中建议新增的内容：

1. `5.x/EOL readiness`：历史材料中 5.x adoption 和服务团队适应成本是关键风险，不应漏掉。
2. `标准交付包`：workflow 优化如果不进入交付包，无法证明能规模化复制。
3. `部署诊断能力`：能降低 G2.1 到 G3.2 阻塞时间，也能支持 IPC 降本后的现场风险控制。

不建议放入 O3 的内容：

- Smart Camera 完整 launch。它更适合 O2。
- 纯质量逃逸指标。它更适合 O1。
- 只写“降低 BOM X%”。Software 无法独自负责 BOM 价格，但可以负责目标配置的软件性能、稳定性、兼容性和可交付性。

## 3. O2/O3 分类视图

| 事项 | 建议归属 | 原因 | KR 形态 |
|---|---|---|---|
| 智能相机 | O2 产品化与市场验证 | 是新产品/新形态，需要证明可 launch、可交付、可复用 | E2E 验证、readiness、FAE 演练 |
| 标准机 | O2 产品化与市场验证，同时连接 O3 | 是标准方案产品化，既要验证市场/场景，也要进入标准交付包 | 标准场景验证、配置时间下降、交付包 |
| IPC 降本 | O3 规模化交付与成本效率 | 核心是低成本配置在 EOL 场景中仍稳定可用 | benchmark、长稳、配置矩阵、Go/No-go |
| workflow 优化 | O3 规模化交付与成本效率 | 核心是减少 FAE/Service 和研发依赖，提高复制效率 | 独立配置成功率、配置耗时下降、模板覆盖 |
| 5.x readiness | O3 规模化交付与成本效率 | 5.x adoption 是软件平台规模化的基础条件 | readiness checklist、风险接受 |
| 部署诊断能力 | O3 initiative，可选 KR | 支撑 IPC 降本和 workflow 进入现场后的问题定位 | 日志/诊断覆盖率、阻塞时间下降 |

## 5. 需要补齐的数据

1. 智能相机 Q3 目标应用场景清单，以及每个场景的软件验收指标。
2. 标准机 Q3 目标标准场景清单，以及基线交付方式、配置耗时、稳定性或复用性指标。

## 7. V1.1 与上一版差异

1. O1 不再重点讨论，只保留质量主线和 IPC freeze 稳定性 KR。
2. O2 从“让 EOL 软件平台更容易被 FAE/Service 独立交付”调整为“智能相机和标准机产品化验证”，更明确承接新产品和标准方案。
3. O3 从“数据驱动的软件研发与产品化闭环”调整为“EOL 平台规模化交付能力和成本效率”，更明确承接 IPC 降本和 workflow 优化。
4. 新增 `5.x/EOL readiness`、`标准交付包`、`部署诊断能力` 作为 O3 的支撑项，因为它们是 IPC 降本和 workflow 优化能否进入现场的必要条件。
