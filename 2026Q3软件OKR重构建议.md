# 2026 Q3 软件团队 OKR 重构建议

生成日期：2026-06-09  
范围：仅聚焦 Software；AI、Hardware、Advanced Technology 暂不展开。  
输入来源：当前 `Engineering OKRs` Google Doc、Google Doc 评论、本地历史报告 `unitx-okr-diagnostic-report.md`、上一版工程级重构草案。  
边界：本文件是本地草案，不写回 Google Doc；所有 `X/Y/Z` 需要用 Jira、现场问题台账、release report、QA 缺陷数据和项目阶段台账补齐。

## 1. 重新调研后的判断

当前 Google Doc 中 Software 部分是：

- O1: Improve EOL software quality and reliability
- O2: Reduce IPC BOM for EOL product without sacrificing reliability
- O3: Ensure a successful launch of smart camera

这三个点都重要，但作为 Software OKR 架构有两个问题：

1. O2 和 O3 太像项目列表。IPC 降本、Smart Camera launch 都应作为软件贡献的场景或 initiative，而不是占据整个 Software Objective。
2. Software 的真实职责更宽：主线质量、严重稳定性问题、现场交付效率、自助配置能力、release 质量闭环、产品需求闭环、工程效率。只写 IPC 和 Smart Camera 会让其他核心工作找不到上层归属。

建议 Software Q3 改为 3 个 Objective：

1. 提升 EOL 主线质量，消除 IPC 卡死等严重稳定性风险。
2. 让 EOL 软件平台更容易被 FAE/Service 独立交付。
3. 建立数据驱动的软件研发与产品化闭环。

IPC 降本、IPC 卡死、Smart Camera、No-code Workflow、5.x adoption、质量评估体系都可以保留，但它们应挂在这 3 条主线下面。

## 2. 当前 O1 的调整方向

当前 O1 的前两个 KR 是合理的质量指标：

- Reduce P0/P1/P2 bugs in new EOL software releases reported by field (escaped QA) to X per release.
- Reduce P0/P1/P2 bugs in new EOL software builds reported by QA (escaped Dev) to X per release.

第三条 `Rootcause fix IPC freeze issues` 不能只写成动作，因为现在已经出现 IPC 卡死等严重问题。它应升级为“严重稳定性风险闭环”的结果型 KR，覆盖：

- 已知 IPC 卡死问题是否完成 root cause。
- 修复是否在目标 IPC 配置、目标版本、目标场景验证通过。
- 现场是否有复发。
- 是否形成可复用的监控、日志、诊断和规避机制。

建议不要写“fix IPC freeze issue”这种无法评分的动作，而写成：

> Q3 结束前，已知 IPC 卡死类 P0/P1 问题 100% 完成根因归类和修复/规避方案；目标版本在目标 IPC 配置上通过 `X` 小时长稳和 `Y` 个真实/准真实 EOL 场景验证，验证期内卡死复现率为 0；新发生 IPC 卡死 P0/P1 均在 24 小时内确认 DRI、72 小时内给出 root cause 或可执行规避方案。

## 3. 推荐 Software OKR

### O1. 提升 EOL 主线质量，消除 IPC 卡死等严重稳定性风险

战略意图：Q3 Software 的第一优先级应是让 EOL 主线版本可被 QA、Service 和客户现场信任。IPC 卡死不是普通 bug，它会直接影响现场稳定性、客户信心和后续降本/配置切换，因此应作为 O1 的核心 KR。

| KR                                                                                                    | 类型        | Owner         | 指标定义                                                                                                                              | 数据源                                | Initiative 示例                                      |
| ----------------------------------------------------------------------------------------------------- | --------- | ------------- | --------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------- | -------------------------------------------------- |
| KR1. 新 EOL software release 在 GA 后 30 天内，field reported escaped QA 的 P0/P1/P2 不超过 `X/Y/Z` per release | Committed | June          | 只统计已 GA release；按问题首次出现版本归属；重复问题按同一 root cause 合并                                                                                 | Jira、field issue 台账、release report | release gate、现场问题分级、GA 后 30 天质量复盘                  |
| KR2. 新 EOL software build 在 QA validation 阶段发现的 escaped Dev P0/P1/P2 不超过 `X/Y/Z` per release          | Committed | Kai           | 统计应在 Dev 阶段发现但逃逸到 QA 的缺陷；需标注漏测原因                                                                                                  | QA bug report、Jira、测试报告            | 单元/集成测试补强、critical path 自动化、代码变更风险分级               |
| KR3. IPC 卡死类 P0/P1 问题完成闭环，目标版本验证期内复现率为 0                                                              | Committed | Colin         | 已知 IPC freeze 问题 100% root cause 分类；目标 IPC 配置通过 `X` 小时长稳和 `Y` 个真实/准真实 EOL 场景验证；新发 P0/P1 24 小时定 DRI、72 小时给 root cause 或 workaround | Jira、现场日志、长稳测试、配置矩阵                | IPC freeze root cause、日志/诊断增强、watchdog/恢复机制、目标配置回归 |
| KR4. P0/P1 软件问题从发现到可执行规避方案的中位时间降低到 `X` 小时以内                                                           | Committed | Software Lead | 包含 field 和 QA P0/P1；可执行规避方案可以是 hotfix、rollback、配置规避或明确操作步骤                                                                        | Jira、incident review               | incident 模板、on-call/DRI 机制、版本 rollback playbook    |

不建议放入 O1 的内容：

- IPC BOM 成本降低本身。成本是业务/产品/供应链结果，Software 应负责性能、稳定性和配置可靠性。
- Smart Camera launch 总结果。Software 只能负责软件 readiness 和 E2E workflow，不应独自承诺完整 launch。

### O2. 让 EOL 软件平台更容易被 FAE/Service 独立交付

战略意图：历史材料反复指向自主交付、5.x adoption、No-code Workflow、G2.1 到 G3.2 周期、FAE/SI 自部署。Software 在这里的贡献不是“完成某个功能”，而是降低现场交付对研发的依赖。

| KR                                                                          | 类型           | Owner              | 指标定义                                       | 数据源                                | Initiative 示例                       |
| --------------------------------------------------------------------------- | ------------ | ------------------ | ------------------------------------------ | ---------------------------------- | ----------------------------------- |
| KR1. 选定 `X` 个 EOL 标准场景，FAE/Service 在无研发现场支持下完成软件配置、运行和内部验收                  | Committed    | Software + Service | 场景需在 Q3 前两周冻结；验收包括配置、采图、workflow、结果输出、问题记录 | deployment rehearsal、UAT report    | No-code Workflow、标准配置模板、部署 playbook |
| KR2. 目标 EOL 项目中，因软件配置、版本、工具问题造成的 G2.1 到 G3.2 阻塞天数降低 `X%`                    | Aspirational | Software + Service | 只统计目标项目清单；阻塞原因需由 Service/Software 共同归类     | 项目阶段台账、Jira、Service notes          | 配置工具、版本兼容性检查、问题快速诊断                 |
| KR3. 5.x/EOL 新部署或升级项目的软件 readiness checklist 覆盖率达到 100%，未通过项在进入现场前关闭或明确风险接受 | Committed    | Software PM/Lead   | checklist 覆盖版本、配置、日志、rollback、已知问题、QA 结果   | readiness checklist、release report | 5.x readiness、部署前检查、风险 sign-off     |
| KR4. 标准交付包覆盖 Top `X` 个高频 EOL 软件能力，FAE/Service 独立使用成功率达到 `Y%`                | Aspirational | Product + Software | 标准交付包包括 workflow 模板、配置说明、日志诊断、FAQ、版本约束     | Service feedback、交付包使用记录           | Workflow 模板、标准配置包、现场诊断工具            |

这里可以承接的原 O2/O3 内容：

- IPC 降本：放在 KR3/KR4 的 readiness 和标准配置中，Software 负责降本 IPC 配置是否跑得稳、是否能被服务独立使用。
- Smart Camera：如果 Q3 Software 必须支持 Smart Camera，则只作为一个标准场景或交付包验证场景，不作为整个 O2/O3。
- No-code Workflow：放在 KR1/KR4，重点从“完成 4 个 CV 功能”改成“FAE/Service 能否独立完成配置并通过验收”。

### O3. 建立数据驱动的软件研发与产品化闭环

战略意图：Q2 的流程规范、质量评估体系、工程主导 RRD、Roadmap 机制都有价值，但不能写成“发布文档/建立机制”。Q3 应衡量这些机制是否让软件团队更快识别高价值问题、更快做出 release 决策、更快把现场问题转成产品改进。

| KR | 类型 | Owner | 指标定义 | 数据源 | Initiative 示例 |
|---|---|---|---|---|---|
| KR1. 100% EOL release 在 GA 前完成质量评分和风险评审，GA 后 30 天完成质量复盘；复盘结论进入下一轮 release gate | Committed | Software + QA | 评分必须包含 escaped Dev、escaped QA、field issue、known risk、rollback plan | quality report、release review | 质量评分体系、release review 模板、风险门禁 |
| KR2. Top `X` 个重复出现的软件问题类别完成 root cause 聚类，其中至少 `Y` 个类别的问题量在 Q3 内下降 `Z%` | Committed | Software Lead | 问题类别按 root cause 而非表象归类；例如 IPC freeze、配置兼容、日志不足、workflow 易用性 | Jira、field issue taxonomy、QA report | 缺陷 taxonomy、重复问题复盘、专项修复 |
| KR3. 工程主导进入 product ranking 的需求中，至少 `X` 个明确绑定交付效率、质量或自助部署指标；至少 `Y` 个在 Q3 交付并验证效果 | Aspirational | Software PM/Lead | 不只统计 RRD 数量，必须写清业务假设、目标指标和上线后验证 | Product ranking、RRD、release notes、field feedback | 工程主导 RRD、需求价值模板、上线后效果验证 |
| KR4. P0/P1/P2 软件问题、release 风险、部署 blocker 的 dashboard 覆盖率达到 100%，每周更新，Red 项均有 DRI 和下一步动作 | Committed | Software Ops/Lead | dashboard 不是结果本身，但作为 Q3 管理依赖；Red 项无 DRI 视为未达成 | Jira dashboard、OKR dashboard、weekly review | dashboard、weekly review、风险升级机制 |

这里替代原 O3 的原因：

- “发布 4 份流程规范”太偏 output，不能证明流程有效。
- “技术 roadmap 机制”可以保留，但要看它是否改变了需求排序和资源取舍。
- “AI-driven R&D 3x”暂不适合放在 Software 顶层 OKR，除非能定义清楚作用范围、基线和样本。

## 4. O2/O3 的其他路径对比

如果希望 O2/O3 更贴近当前业务，有三种可选路径：

| 路径                     | 优点                                              | 风险                                  | 适合放在               |
| ---------------------- | ----------------------------------------------- | ----------------------------------- | ------------------ |
| IPC 降本路径               | 与毛利直接相关，Q3 可见度高                                 | 对 Software 来说过窄，且成本 owner 不完全在软件    | O2/KR 或 initiative |
| Smart Camera launch 路径 | 战略产品清晰，跨团队关注度高                                  | Software 无法独自对 launch 结果负责，容易变成项目清单 | O2 的验证场景或独立项目计划    |
| 自主交付/平台化路径             | 覆盖 5.x、Workflow、配置、部署效率、Service 独立性，和历史 OKR 更一致 | 需要 Service/Product 提供项目台账和验收数据      | 推荐作为 O2            |
| 研发与质量闭环路径              | 覆盖流程、质量评估、RRD、Roadmap、dashboard，能解决历史 OKR 口径问题  | 容易写成“建机制”，必须绑定问题下降和决策效率             | 推荐作为 O3            |

推荐选择：

- O2 使用“自主交付/平台化路径”。
- O3 使用“研发与质量闭环路径”。
- IPC 降本和 Smart Camera 分别作为 O2 的验证场景或 initiative，不单独占据 Objective。

## 6. 需要补齐的数据

1. Q2 每个 EOL release 的 field escaped QA P0/P1/P2 数量。
2. Q2 每个 EOL build 的 escaped Dev P0/P1/P2 数量。
3. IPC 卡死问题清单：客户/场景、IPC 配置、软件版本、日志、复现路径、当前 root cause 假设、修复状态。
4. 目标 EOL 标准场景清单：哪些场景要用于 FAE/Service 独立交付验证。
5. G2.1 到 G3.2 阶段台账，以及其中由软件造成的阻塞天数。
6. 5.x/EOL 新部署或升级项目清单，以及进入现场前的软件 readiness 标准。
7. Top 重复软件问题类别的 Q2 baseline。
8. 工程主导 product/RRD 的现有清单和上线后效果验证数据。

## 7. 拆解到下一层的方法

每条 KR 拆到下一层时，不要按项目名直接分配，而按责任面拆：

| 上层 KR      | 下一层拆解                                                               |
| ---------- | ------------------------------------------------------------------- |
| O1 KR1/KR2 | QA escaped defect、Dev escaped defect、release gate、critical path 自动化 |
| O1 KR3     | IPC freeze root cause、长稳测试、日志诊断、watchdog/恢复、配置回归                    |
| O1 KR4     | incident response、hotfix/rollback、on-call DRI、问题复盘                  |
| O2 KR1/KR4 | No-code Workflow、标准配置模板、部署 playbook、FAE enablement 验证               |
| O2 KR2/KR3 | 5.x readiness、版本兼容检查、G2.1-G3.2 software blocker 消除                  |
| O3 KR1/KR2 | 质量评分体系、field issue taxonomy、重复问题专项下降                                |
| O3 KR3/KR4 | 工程主导 RRD、product ranking、OKR/dashboard、风险升级                         |

这样下一层团队或个人不会只拿到“做 IPC 降本”或“支持 Smart Camera”这种项目任务，而是能看到自己负责的结果指标、数据源和验收方式。
